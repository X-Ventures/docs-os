## 2. Context

### What Is Bittensor?

Bittensor is a **decentralized machine learning network** built on the TAO token. It operates as a network of specialized subnets, each focused on a different AI task (text generation, image generation, data scraping, inference, etc.).

- **Token:** TAO
- **Supply cap:** 21,000,000 TAO (Bitcoin-like issuance curve)
- **Current supply:** ~7.5M TAO
- **Consensus:** Yuma Consensus (reputation-weighted)
- **Emission split:** 41% Validators / 41% Miners / 18% Subnet Owners

### Validator Role

Validators are the **quality control layer** of Bittensor. They:

1. Stake TAO to register on subnets
2. Send tasks to miners and evaluate responses
3. Set weights that determine miner rewards
4. Earn 41% of subnet emissions proportional to their stake and performance
5. Can set a **take rate** (0-18%) on delegated stake

A validator's earnings depend on: stake size, number of subnets validated, VTrust score (consensus alignment), and subnet emission allocation.

## 3. Strategy Comparison

### Strategy A — Weight Copying

**Approach:** Copy weights from a top-performing validator (e.g., Foundry, Taostats) instead of running real evaluation tasks.

| Parameter | Value |
| --- | --- |
| OPEX | €2,100/mo |
| Setup Time | 1-2 days |
| APY (estimated) | ~4.5% |
| ML Expertise Required | None |
| Infrastructure | CPU server only (Hetzner AX102) |

**Pros:**
- Fastest time to yield
- Minimal technical complexity
- Low infrastructure cost

**Cons:**
- **Commit Reveal risk** — subnet owners can enable Commit Reveal, which makes weight copying impossible. This is a protocol-level toggle that can be activated at any time.
- Lower APY than real validation (weight copiers get penalized by VTrust scoring)
- No competitive moat — anyone can weight copy
- Reputation risk if identified as a weight copier

---

### Strategy B — Real Validation

**Approach:** Run actual validation tasks — send challenges to miners, evaluate responses, set weights based on real performance metrics.

| Parameter | Value |
| --- | --- |
| OPEX | €4,800/mo |
| Setup Time | 2-4 weeks |
| APY (estimated) | ~5.5% |
| ML Expertise Required | High (ML engineer or equivalent) |
| Infrastructure | CPU server + GPU cloud (A100/H100 instances) |

**Pros:**
- Highest yield potential
- Immune to Commit Reveal
- Builds real expertise and network position
- Option value for subnet ownership

**Cons:**
- High OPEX — GPU compute is expensive
- Requires dedicated ML/infra operator
- Longer ramp-up period
- Higher complexity = more failure modes

---

### Strategy C — Hybrid (RECOMMENDED)

**Approach:** Start with weight copying for immediate yield. Transition to real validation on 2-3 high-value subnets over 3-6 months.

| Parameter | Value |
| --- | --- |
| OPEX | €3,200/mo |
| Setup Time | 1-2 days (WC), then gradual |
| APY (blended estimate) | ~5.0% |
| ML Expertise Required | Medium (hire within 3 months) |
| Infrastructure | CPU server + selective GPU usage |

**Timeline:**

| Phase | Period | Activity | OPEX |
| --- | --- | --- | --- |
| Phase 1 | Month 1-3 | Weight copying on 5-8 subnets | €2,100/mo |
| Phase 2 | Month 3-6 | Transition 2-3 subnets to real validation | €3,200/mo |
| Phase 3 | Month 6-12 | Full real validation on priority subnets | €4,200/mo |

**Pros:**
- Immediate yield from day 1
- Time to hire and train
- Gradual OPEX ramp (cash flow friendly)
- Hedged against Commit Reveal (partial)

**Cons:**
- Still exposed to Commit Reveal on WC subnets during Phase 1-2
- Requires discipline to actually transition (easy to stay on WC)

## 6. Risk Register

### Technical Risks

| # | Risk | Severity | Likelihood | Impact | Mitigation |
| --- | --- | --- | --- | --- | --- |
| T1 | Server downtime (missed blocks) | High | Medium | Missed emissions, VTrust decay | Redundant servers, auto-restart, 24/7 monitoring |
| T2 | GPU instance unavailability | Medium | Medium | Cannot run real validation | Multi-cloud strategy (Hetzner + Lambda + RunPod) |
| T3 | Software bugs in validator code | Medium | High | Incorrect weights, slashing risk | Staging environment, automated testing, gradual rollout |
| T4 | Network connectivity issues | Medium | Low | Temporary emission loss | Dual ISP, failover networking |

### Market Risks

| # | Risk | Severity | Likelihood | Impact | Mitigation |
| --- | --- | --- | --- | --- | --- |
| M1 | TAO price crash (-40% or more) | Critical | Medium | Portfolio loss overwhelms yield | Stop-loss at -30%, position sizing, don't over-allocate |
| M2 | APY compression from increased staking | Medium | High | Lower per-validator yield | Scale stake to maintain share, optimize subnet selection |
| M3 | Exchange liquidity issues | Low | Low | Difficulty exiting position | Use multiple exchanges, OTC for large positions |

## 7. Go / No-Go Checklist

### Infrastructure Prerequisites

- [ ] Hetzner AX102 server provisioned and tested
- [ ] Bittensor node synced and operational
- [ ] Monitoring stack deployed (Grafana + Prometheus + PagerDuty)
- [ ] Backup and failover procedures documented and tested

### Stake Prerequisites

- [ ] Minimum 5,000 TAO acquired (or committed)
- [ ] Cold wallet + hot wallet architecture configured
- [ ] Multi-sig or equivalent custody solution for >€500K

### Expertise Prerequisites

- [ ] Validator operator identified (internal or contractor)
- [ ] Runbook for weight copying written and tested
- [ ] Incident response plan documented
- [ ] ML engineer pipeline started (for Phase 2 transition)

### Activation Thresholds

- [ ] TAO price above $150 (below = re-evaluate thesis)
- [ ] Network emissions stable (no pending protocol changes that alter economics more than 20%)
- [ ] At least 3 target subnets confirmed operational and accepting new validators

### Stop-Loss Conditions

- [ ] TAO drops below $120 sustained for 14 days → reduce exposure
- [ ] Operational losses exceed €10,000/month for 3 consecutive months → exit weight copying
- [ ] Commit Reveal activated on over 50% of target subnets before Phase 2 transition: emergency pivot to real validation or exit
